[Heading 1] 论文选题的意义及背景
[Heading 2] 研究背景
[论文正文] 三维重建是计算机视觉中的基础问题，目标是根据图像、视频或多视图观测恢复场景的三维结构，在机器人导航、增强现实、数字内容制作和三维场景理解等领域都有实际用途。传统三维重建通常包括特征提取与匹配、相机位姿估计、稀疏重建、稠密匹配和点云融合等步骤，典型工具包括 COLMAP 等。这类方法在纹理充足、视角变化适中、相机运动稳定的场景中表现较好，但在弱纹理、动态物体、宽基线图像和未标定输入下，容易出现匹配失败或重建不完整的问题。
[论文正文] 近几年，前馈式三维重建模型逐渐受到关注。以 DUSt3R 为代表的方法尝试直接从图像中预测点图、深度或相机相关几何表示，降低了对传统几何流程和相机标定信息的依赖。后续 MASt3R、MonST3R、Spann3R、Fast3R、CUT3R 等模型又分别在图像匹配、动态场景、多视图输入、长序列处理和在线重建等方向扩展，使三维重建实验有了更多可选工具。
[论文正文] 但在实际学习和实验过程中，多模型比较的困难更多来自工程使用环节。不同模型需要分别配置环境、准备权重、调整脚本、上传数据、运行远端 GPU 命令并下载结果；输出内容也可能是点图、深度、mask、相机参数或可视化文件，文件组织方式并不统一。如果只保留最终截图，很难追溯模型在固定输入上的真实表现和失败原因。
[论文正文] 因此，本课题拟设计一个面向前馈式三维重建模型的聚合管理平台。课题重点不在重新提出三维重建网络，而是围绕模型实验流程，建立统一的模型描述、任务提交、远端运行、结果回传和指标记录方式，为后续复现、横向比较和项目展示提供基础工具。
[Heading 2] 本课题的核心价值意义
[论文正文] 基于上述背景可以看出，前馈式三维重建模型发展较快，但模型之间的工程接口并不统一。对学生实验和项目开发而言，影响效率的往往是环境配置、输入准备、运行脚本、结果查找和失败复盘等重复环节。本课题希望把这些环节整理为相对统一的平台流程，使模型复现和对比更加可追溯。
[论文正文] 本课题的意义主要体现在三个方面：一是通过模型注册和配置文件统一记录不同模型的输入、输出、权重、运行命令和资源需求；二是通过任务提交、远端执行和结果同步流程，降低本地运行大模型的压力；三是通过结果元数据和指标表保存成功样例、失败样例、日志和可视化产物，为后续报告和论文材料提供依据。
[Heading 1] 国内外研究现状分析
[论文正文] 围绕三维重建，国内外研究已经形成了较为丰富的技术体系。从传统几何方法到深度学习方法，再到近年来的前馈式重建模型，各类方法在输入条件、输出形式和工程使用方式上差异明显。本课题关注的是前馈式三维重建模型的聚合管理，因此需要结合算法特点分析平台需求。
[Heading 2] 基于几何的传统三维重建方法
[论文正文] 传统三维重建主要依赖多视图几何。SfM 方法通过图像特征匹配估计相机位姿和稀疏点云，MVS 方法在已知相机位姿的基础上恢复稠密深度或点云。COLMAP 等工具把这些步骤组织成较成熟的流程，在静态场景和高质量图像中有较强可用性。这类方法的优势是几何约束清楚、结果可解释，且有较长时间的工程积累。
[论文正文] 其局限主要体现在输入条件要求较高。若图像之间重叠区域不足、纹理信息弱、物体发生运动或相机轨迹不稳定，特征匹配和位姿估计容易失败。对于初学者或多模型实验场景，传统流程还需要较多参数设置和中间结果判断，不利于快速比较不同方法的实际效果。
[Heading 2] 基于学习的深度估计与多视图重建方法
[论文正文] 深度学习方法在深度估计、多视图立体和场景重建中取得了较多进展。MVSNet 等方法利用神经网络构建立体匹配代价体，NeuralRecon 等方法结合 RGB-D 或序列信息进行场景级重建，NeRF 及其后续方法则从隐式表示角度建模场景。这类方法在特定任务上效果较好，但往往需要相机参数、深度监督或较长优化时间，不一定适合直接做快速批量比较。
[Heading 2] 前馈式三维重建模型
[论文正文] 前馈式三维重建模型试图减少传统流程中的显式特征匹配和手工几何步骤。DUSt3R 能够在未标定图像对或多视图条件下直接输出点图；MASt3R 强化了图像匹配与三维几何约束之间的联系；MonST3R 面向动态场景和视频输入；Spann3R、Fast3R 和 CUT3R 等工作分别关注长序列、多图输入、运行效率和在线重建等问题。
[论文正文] 这些方法更接近端到端模型推理，使用时通常只需要准备图像、视频或少量参数即可得到几何结果。但它们的依赖版本、权重路径、CUDA 扩展、显存限制和输出格式差异较大，直接比较时容易遇到格式转换和指标口径问题。

[Heading 2] 实验管理工具与平台需求
[论文正文] 现有开源工具多数服务于单一算法或传统重建流程。COLMAP 有较完整的命令行和图形界面，但主要面向几何重建；深度学习模型通常提供独立推理脚本和 demo，方便复现论文示例，却较少考虑多个前馈式三维重建模型在同一输入和同一记录格式下的比较。
[Heading 1] 前期准备、研究目标与研究内容
[Heading 2] 前期准备与已有工作
[论文正文] 开题阶段已经完成了相关模型族的初步阅读，重点整理 DUSt3R、MASt3R、MonST3R、Spann3R、Fast3R、CUT3R 和 Align3R 等方法的输入形式、输出表示、适用场景和可能失败条件。
[论文正文] 模型失败条件也进行了初步整理。例如，静态图像对模型可能在低纹理、视角差过大或重叠区域不足时表现不稳定；动态场景模型可能受到运动模糊、遮挡和长序列显存限制影响；部分模型在 Windows 本地直接运行困难，需要远端 Linux GPU 环境或额外 CUDA 扩展。
[论文正文] 在平台方案上，已经形成了基本框架：桌面前端用于选择模型、上传输入和查看结果；本地后端用于保存任务、校验配置和提供 API；远端调度通过 SSH/SCP 上传输入、启动命令并同步产物；模型执行器封装具体模型的运行脚本。
[论文正文] 在接口和文件格式方面，开题阶段拟定了 job.json、status.json 和 scene_meta.json 三类关键状态文件，分别记录任务配置、运行状态和结果产物。这些文件将作为任务复现和结果对比的依据。
[Heading 2] 研究目标
[论文正文] 本课题计划完成一个可运行的前馈式三维重建模型实验管理原型，重点目标如下：
[论文正文] 1、建立模型注册与描述方式，形成模型配置文件、字段说明和校验脚本。
[论文正文] 2、实现任务提交与远端运行流程，完成任务 API、SSH/SCP 调用逻辑和 runner 脚本。
[论文正文] 3、建立结果解析与产物管理方式，形成 scene_meta.json 规范、结果目录结构和可视化入口。
[论文正文] 4、设计多模型对比实验，形成样本清单、指标表、对照模型列表和失败记录表。
[论文正文] 5、整理接口说明、实验设计说明、运行记录和阶段报告，方便后续复现实验过程。
[Heading 2] 研究内容
[Heading 3] 模型注册与配置模块
[论文正文] 该模块将不同 3R 模型的论文信息、代码仓库、运行环境、权重路径、输入要求和输出格式整理为统一配置文件，并提供字段校验。输出为平台可读取的模型注册表，重点解决模型信息分散的问题。
[Heading 3] 任务提交与远端执行模块
[论文正文] 该模块根据用户选择的模型、输入数据和参数生成 job.json，通过 SSH/SCP 上传到远端目录，再调用对应 runner 执行推理。运行过程中记录日志和 status.json，完成后同步输出目录。
[Heading 3] 结果解析与展示模块
[论文正文] 该模块根据模型输出整理 scene_meta.json，记录点图、深度、mask、相机信息、图片或视频产物的路径。前端根据元数据展示结果，使不同模型输出能够归到同一任务目录下。
[Heading 3] 指标评估与对比模块
[论文正文] 该模块根据固定样本的多模型输出进行评估。有真值数据时计算深度误差、相机轨迹误差和点云指标；无真值样本时记录运行耗时、成功率、输出完整性、可视化质量和失败原因。
[Heading 3] 文档与实验记录模块
[论文正文] 该模块整理接口文档、运行说明、模型接入说明和实验记录模板，输出项目文档、阶段报告和后续论文材料。
[Heading 2] 拟解决的关键问题
[论文正文] 1、不同模型的运行环境、输入格式和输出结构差异较大，平台需要在不强行改动模型仓库的前提下完成统一接入。
[论文正文] 2、远端 GPU 任务存在上传、运行、日志回传和异常中断等环节，平台需要保证每一步都有状态记录，避免任务失败后无法追溯。
[论文正文] 3、不同模型产物不完全一致，部分指标无法直接计算，因此需要设计可选字段和失败原因记录，而不是简单要求所有模型输出完全相同。
[论文正文] 4、实验结果需要兼顾定量指标和实际可用性，既记录有真值数据集上的误差，也记录无真值样例中的运行耗时、输出完整性和失败类型。

[Heading 1] 研究方案及可行性分析
[Heading 2] 平台总体技术路线
[论文正文] 本研究拟采用本地桌面端加远端 GPU 执行的总体方案。本地侧负责模型管理、任务创建、状态展示和结果查看，远端侧负责模型推理和产物生成，二者通过 SSH/SCP 完成输入上传、命令启动和结果同步。
[论文正文] 本课题的技术路线可以概括为：输入数据 -> 模型选择与参数配置 -> 生成 job.json -> 后端校验 -> SSH/SCP 上传 -> 远端 runner 执行 -> 生成 status.json 和日志 -> 生成 scene_meta.json -> 本地同步 -> 可视化与指标评估。
[论文正文] 平台技术栈拟采用 Tauri 2 与 React 作为桌面前端，FastAPI 作为本地后端，Python runner 作为模型执行器，SSH/SCP 作为远端传输方式。该路线便于先完成小规模原型，再逐步补充更多模型和指标。
[Heading 2] 核心模块设计
[论文正文] 模型注册模块采用配置文件保存模型名称、论文或仓库地址、运行环境、权重路径、输入类型、输出类型、默认命令、显存需求和已知限制。平台启动时读取注册表，并在前端展示可用模型。
[论文正文] 任务管理模块负责生成 job.json 和任务目录。任务提交后，本地后端将输入和配置上传到远端目录，远端 runner 根据模型配置启动推理脚本，并持续写入 status.json 和日志。
[论文正文] 结果解析模块负责整理 scene_meta.json，记录点图、深度图、mask、相机参数、可视化图片、视频和日志等产物路径。若某个模型没有输出某类文件，则对应字段为空并记录原因。
[Heading 2] 实验设计
[Heading 3] 数据集与样本选择
[论文正文] 实验计划分为小样本功能验证和公开数据集对比两部分。小样本用于检查平台能否正常提交任务、运行模型和回传结果；公开数据集用于进一步比较不同模型的输出质量和可用性。
[论文正文] 公开数据集拟参考 DTU、ETH3D、Tanks and Temples、ScanNet、7-Scenes 等，动态或视频输入可选取公开视频数据和自建短序列样例。具体数据集将根据模型支持的输入格式、下载成本和真值可用性进行筛选。
[Heading 3] 对照模型
[论文正文] 前期计划以 DUSt3R 作为基础模型进行流程验证，再逐步接入 MASt3R、MonST3R、Spann3R、Fast3R、CUT3R 等模型。若时间允许，可加入 COLMAP 作为传统几何方法参考。
[Heading 3] 评价指标
[论文正文] 实验指标按数据条件分为两类。有真值数据时，拟采用 AbsRel、RMSE、ATE、RPE、Chamfer Distance、F-score 等指标；无真值样本时，记录运行耗时、任务成功率、输出完整性、可视化结果、错误日志和显存限制等信息。
[Heading 3] 实验类型
[论文正文] 计划开展功能验证、多模型对比、参数影响和失败案例分析四类实验，分别检查平台流程是否跑通、不同模型在相同样本上的表现、输入参数变化带来的影响，以及失败任务的原因分类。
[Heading 2] 可行性分析
[论文正文] 从技术基础看，Tauri、React、FastAPI、Python 和 SSH/SCP 都是成熟工具，适合搭建本地桌面端和远端执行流程。前期已经初步整理了模型族、接口格式和 runner 设计，可以支持原型开发。
[论文正文] 从数据和模型来源看，DUSt3R、MASt3R 等模型已有公开论文和代码，部分数据集也提供公开样本和真值。后续可以先从小样本验证开始，再扩展到公开数据集。
[论文正文] 主要风险包括模型环境复杂、输出格式不一致、真值与模型输出存在尺度或坐标系差异，以及时间有限难以完成全部模型实验。应对方式是优先接入环境稳定的模型，先定义产物索引和可选字段，并从小规模样本逐步扩展。
[Heading 1] 预期达到的目标和研究成果
[论文正文] 本课题预期完成一个面向前馈式三维重建模型的实验管理原型，包括前端界面、本地后端和远端 runner。平台能够完成模型查询、输入选择、任务创建、远端运行、日志保存、结果同步和基本结果查看。
[论文正文] 在模型接入方面，预期形成若干模型配置文件，记录 DUSt3R、MASt3R、MonST3R、Spann3R、Fast3R、CUT3R 等模型的输入、输出、环境和运行方式。
[论文正文] 在实验记录方面，预期形成 job.json、status.json、scene_meta.json 等任务记录格式，并整理固定实验样本、基础指标脚本、对比表格和失败案例记录表。
[论文正文] 在文档和论文材料方面，预期完成项目说明文档、接口文档、模型接入说明、阶段报告和后续论文材料。

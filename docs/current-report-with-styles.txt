[目录摘要] 摘要
[论文正文] 本项目围绕前馈式三维重建模型的实验管理问题展开。刚开始接触这些模型时，最直接的感觉是论文和代码都不少，但真正要把模型跑起来、把输入输出整理清楚，并不比理解算法本身轻松。不同模型的环境、权重、运行脚本和结果目录差异很大，如果只靠手工命令和文件夹记录，后面很难说明一次实验具体用了什么输入、什么参数、在哪一步失败、最后产物是否完整。
[论文正文] 针对这个问题，本项目设计并实现了 3R All-in-One 多模型三维重建实验管理平台。平台不重新提出新的三维重建网络，而是把模型登记、任务创建、远端运行、日志回传、结果检查和实验记录这些工程环节整理到一个本地桌面软件中。桌面端负责交互和展示，本地后端负责任务状态和模型信息管理，远端 GPU 服务器负责实际模型推理。
[论文正文] 结题阶段，平台已经完成 DUSt3R、MASt3R、MonST3R、Spann3R、Fast3R、Align3R、CUT3R 和 Dream3R 等模型的登记与基础接入工作。其中 Dream3R v1.1 按候选几何融合模型接入，支持 synthetic 演示模式和 proposal-cache 输入模式。系统已经生成 Windows 桌面稳定版，并通过前端构建、桌面打包和模型蓝图校验。演示素材选取 ETH3D delivery_area 场景中的 12 张图像，用于展示从输入选择到任务结果检查的完整流程。
[论文正文] 关键词：三维重建；前馈式模型；实验管理；远端运行；Dream3R；桌面软件
[Heading 1] 绪论
[Heading 2] 课题背景
[论文正文] 三维重建的目标是从图像、视频或多视角观测中恢复场景的三维结构。这个方向在机器人导航、增强现实、数字内容制作和空间理解中都有实际用途。传统三维重建一般要经过特征匹配、相机位姿估计、稀疏点云、稠密深度和融合等步骤，工具链已经比较成熟，但对图像质量、视角重叠和相机运动稳定性有较高要求。
[论文正文] 近几年，前馈式三维重建模型发展很快。以 DUSt3R 为代表的方法尝试直接从图像中预测点图、深度或相机相关几何表示，减少对传统几何流水线和相机标定的依赖。后续 MASt3R、MonST3R、Spann3R、Fast3R、CUT3R 等模型又分别在图像匹配、动态场景、多图输入、长序列处理和在线重建上继续扩展。
[论文正文] 这些模型给课程项目提供了很好的实验对象，但实际复现时也带来一个很现实的问题：模型越多，工程管理越乱。每个模型都有自己的仓库、conda 环境、权重路径、运行命令和输出格式。手工跑一个模型还能勉强记录，多个模型一起比较时，就很容易出现输入混乱、日志丢失、输出找不到、失败原因说不清的情况。
[Heading 2] 研究现状与项目定位
[论文正文] 现有三维重建工具大致可以分为两类。一类是 COLMAP 等传统几何工具，流程完整，结果可解释，但主要面向几何重建本身。另一类是各个深度学习模型的开源代码，它们通常方便复现论文示例，却很少照顾多个模型在同一平台中的统一运行和对比。
[论文正文] 本项目的定位不是再做一个新的模型，也不是替代这些开源仓库，而是做实验管理软件。它要解决的是学生在复现和比较模型时经常遇到的工程问题：模型信息怎么登记，任务怎么创建，远端怎么运行，输出怎么回传，日志和参数怎么保存，结果是否完整怎么检查。
[论文正文] 这种定位比较贴近本课题的实际情况。结题展示时，单独放几张效果图并不能说明完整工作量；一个能创建任务、回传结果、检查日志和保存记录的软件，更能说明项目确实把实验流程连起来了。
[Heading 2] 研究内容与目标
[论文正文] 本项目主要完成四方面工作。第一，整理前馈式三维重建模型的输入、输出和运行要求，形成平台可以读取的模型登记信息。第二，完成本地桌面软件和本地后端，使用户可以在软件中选择模型、上传输入并创建任务。第三，打通远端 GPU 服务器运行流程，通过 SSH/SCP 完成输入上传、命令执行、日志保存和结果回传。第四，接入 Dream3R，并生成可以用于结题演示的 Windows 稳定版软件。
[论文正文] 项目目标不是完整复现每篇论文的全部 benchmark，而是完成一个可运行、可演示、后续可扩展的平台原型。对当前阶段来说，只要能把不同模型的实验流程统一管理起来，并留下清楚的任务记录和结果依据，就已经达到了结题工作的核心目标。
[Heading 1] 相关理论与基础
[Heading 2] 三维重建流程基础
[论文正文] 传统三维重建通常从多视图几何出发。SfM 方法先通过图像特征匹配估计相机位姿和稀疏点云，MVS 方法再根据多视图一致性恢复稠密深度或点云。这类方法的优点是几何约束明确，结果比较容易解释；缺点是对输入条件敏感，弱纹理、重复纹理、运动模糊和视角差过大时容易失败。
[论文正文] 从实验管理角度看，三维重建结果不只是最后的点云或深度图。一次实验还包括输入图像、模型版本、运行参数、相机信息、日志、中间文件和结果索引。缺少这些记录时，即使最后有一张看起来不错的可视化图片，也很难复查这次实验是怎么得到的。
[Heading 2] 前馈式三维重建模型
[论文正文] 前馈式三维重建模型把一部分传统几何步骤放到神经网络前向推理中完成。DUSt3R 可以在未标定图像条件下预测点图；MASt3R 强化了图像匹配和三维几何之间的联系；MonST3R 面向动态场景；Spann3R 和 CUT3R 更关注序列状态；Fast3R 则强调多图输入下的推理效率。
[论文正文] 这些模型使用起来看似更直接，但工程差异并没有消失。它们的输入可能是图像对、多视图图像、视频片段或缓存文件，输出可能是点图、深度、mask、相机参数、可视化图片或 numpy 文件。平台如果强行要求所有模型输出完全一致，反而不现实。因此本项目采用“统一任务记录，允许产物差异”的方式来管理这些模型。
[Heading 2] 软件平台基础
[论文正文] 本项目采用本地桌面端、本地后端和远端服务器三部分结构。桌面端用于用户交互，本地后端用于保存任务状态、模型信息和结果索引，远端 GPU 服务器用于运行真正的模型推理。这样的分工能避免把复杂模型环境全部搬到 Windows 本地，也方便在普通电脑上进行演示。
[论文正文] 任务运行过程中，平台需要记录几个关键信息：任务由哪个模型创建，输入文件有哪些，运行参数是什么，任务处于什么状态，远端日志是否正常，输出目录里是否有必需产物。只有这些信息都能查到，后续才谈得上复现和比较。
[Heading 2] Dream3R 接入基础
[论文正文] Dream3R 在本项目中不是普通的图片输入重建模型。它更适合作为候选几何融合模型来看待，正式输入可以是 proposal-cache，也就是由上游模型或缓存文件提供的候选几何结果。为了保证结题演示稳定，平台同时保留 synthetic 模式，用于不依赖外部大文件的通路演示。
[论文正文] 因此，Dream3R 的接入重点不是把 ETH3D 图片直接丢给它就结束，而是让平台能够识别它的输入类型、参数、运行方式和输出产物。当前版本输出的结果记录包括任务摘要、Dream3R 报告和可选的融合产物，便于在结果检查页面查看。
[Heading 1] 系统设计与实现
[Heading 2] 总体设计
[论文正文] 系统采用“本地负责管理，远端负责计算”的设计。本地软件处理用户能看到的部分，包括模型选择、任务创建、任务队列、结果检查和配置管理；本地后端维护任务状态、模型登记和结果索引；远端服务器保存模型仓库、权重和运行环境，并负责实际推理。
[论文正文] 一次完整实验大致经过以下流程：用户在软件中选择模型和输入，平台生成任务并保存配置；后端把输入和运行信息上传到远端；远端 runner 启动模型并写入日志；任务完成后，本地同步输出目录；用户在任务详情中查看结果、日志和产物完整性。这个流程把原来分散在命令行和远端文件夹里的操作收到了一个软件里。
[Heading 2] 任务流程设计
[论文正文] 任务创建时，平台根据所选模型限制输入类型和参数。比如静态重建模型主要接收图像，动态模型可能接收视频或序列，Dream3R 则支持 synthetic 演示或 proposal-cache 输入。这样做可以减少明显不合法的任务提交。
[论文正文] 任务运行时，平台重点保存状态和日志。任务如果成功，用户可以看到输出产物和结果索引；任务如果失败，用户至少能看到失败发生在哪个阶段，而不是只得到一个空文件夹。对实验管理来说，失败记录同样有价值，因为它能说明模型在什么输入或环境下不稳定。
[Heading 2] 模型接入方式
[论文正文] 新模型接入时，本项目没有把模型逻辑直接写死在页面中，而是拆成几类信息：模型说明、输入类型、运行参数、远端执行脚本和输出要求。前端只根据后端返回的模型信息生成选择项和参数表单，具体模型怎么运行由后端和 runner 处理。
[论文正文] 这种方式的好处是边界比较清楚。以后继续接入模型时，不需要重做整个软件，只需要补充对应模型的登记信息、运行方式和输出检查规则。对于结题阶段，这种设计已经能够支撑多个代表模型共用同一个任务管理流程。
[Heading 2] 软件使用流程
[论文正文] 从用户角度看，软件的使用流程比较直接。打开软件后先查看已有任务和模型状态；需要新实验时，在新建任务页面选择模型、输入文件和参数；任务提交后回到队列观察运行状态；任务结束后进入结果检查页面查看产物、日志和输出完整性。
[论文正文] 这个流程对应的不是单纯页面堆叠，而是实际实验过程的整理。过去需要在本地文件夹、远端终端、脚本参数和结果目录之间来回切换，现在至少可以在同一个软件里追踪一次实验从创建到结果查看的过程。
[Heading 2] Dream3R 接入实现
[论文正文] Dream3R v1.1 在平台中按候选几何融合流程接入。平台为它保留了 demo_mode、domain、batch、views、patches、d_memory 和 device 等参数，其中 synthetic 模式用于稳定演示，cache 模式用于上传 .pt 或 .pth proposal-cache 文件。
[论文正文] 任务完成后，Dream3R 的结果会被整理为平台能识别的结果记录。当前版本不把它夸大成完整数据集 benchmark，而是明确写成平台接入和演示通路：能创建任务，能调用 runner，能回收结果，能在软件里检查产物。这个表述更符合目前实际完成情况。
[Heading 1] 测试与结果分析
[Heading 2] 软件完成情况
[论文正文] 桌面端可以完成模型选择、任务创建、任务队列查看、结果检查和系统配置；本地后端可以保存任务状态、提供模型信息、执行远端调度并整理结果索引；远端服务器负责模型推理和输出生成。
[论文正文] 结题前重点修复了影响演示的问题。辅助评估配置页面曾经因为前后端字段不一致出现白屏，修复后可以正常打开和保存配置。结果检查页面原来内容分栏过挤，调整后按任务事实、产物证据、分析结果和日志分区显示，长日志也可以正常换行。
[Heading 2] 模型接入结果
[论文正文] 当前平台已经登记并整理了 DUSt3R、MASt3R、MonST3R、Spann3R、Fast3R、Align3R、CUT3R 和 Dream3R。不同模型的完成程度并不完全相同，因此报告中不把所有模型都写成完整数据集验证。DUSt3R 作为静态重建基础路线，MASt3R、Spann3R、Fast3R 和 CUT3R 主要完成 smoke 级验证，MonST3R 有标准样本路径验证，Align3R 保留为 runner 就绪状态，Dream3R 完成 v1.1 synthetic/cache 演示通路接入。
[Heading 2] 结果分析
[论文正文] 从完成情况看，项目已经解决了最初设想中的主要问题：模型信息不再完全散在各个仓库里，任务运行不再只靠手工记录，结果产物不再只是凭猜测，日志和输出检查也进入了软件流程。对一个课程结题项目来说，这比单纯堆算法名更有实际意义。
[Heading 1] 总结
[论文正文] 本项目完成了一个面向前馈式三维重建模型的本地实验管理平台。相比开题时的设想，结题版本已经把模型登记、任务创建、远端运行、日志回传、结果检查、实验记录和桌面打包连成了可以演示的流程。
[论文正文] 在模型接入方面，平台整理了多种代表性 3R 模型，并完成 Dream3R v1.1 的接入。Dream3R 在本项目中被明确定位为候选几何融合模型，synthetic 模式用于稳定演示，proposal-cache 模式用于后续连接上游几何结果。
[论文正文] 在软件交付方面，项目已经生成稳定版客户端，并通过前端构建、桌面打包和模型蓝图校验。当前结果说明，本项目已经从最初的想法推进到了可运行、可检查、可继续扩展的软件原型。后续如果继续做，重点应放在固定样本对比、更多结果预览和更完整的定量评估上。
